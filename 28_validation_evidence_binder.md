# 验证证据包与审计规则

更新时间：2026-05-19

## 目的

这份文件定义真实验证阶段如何留证据。目标是让最终进入判断不是“看起来合理”，而是能追溯到：

1. 哪个服务商说了什么。
2. 哪个真实门店在哪个 AI 入口下出现了什么回答。
3. 哪些原始回答、截图、访谈原话支撑了 Go/No-Go。
4. 哪些结论还只是桌面假设，不能当成已验证事实。

## 证据分级

| 等级 | 类型 | 能证明什么 | 不能证明什么 |
| --- | --- | --- | --- |
| L0 桌面资料 | 公开报道、官网、产品页、行业报告 | 生态存在、工具范式、公开趋势 | 不能证明服务商愿意付费 |
| L1 访谈反馈 | 服务商/运营/商家访谈原话 | 需求、预算归属、试讲意愿、反对理由 | 不能证明真实 AI 回答存在差异 |
| L2 真实门店测试 | 原始 AI 回答、截图、评分表、复测记录 | 门店/竞品差异、信息错误、可转任务 | 不能证明商家愿意买单 |
| L3 商家/服务商试讲 | 服务商拿报告给商家后的反馈 | 报告是否能进入销售、月报或续费 | 不能证明规模化复购 |
| L4 付费试点 | 付款记录、试点合同、连续使用记录 | 真实付费意愿和早期商业化 | 不能自动证明大规模市场 |

当前项目大多数材料处于 L0，部分为 L1/L2 的准备模板。goal 不能标记完成，直到至少有 L1 和 L2 的真实证据补齐，并能修订最终进入判断。

## 证据索引

统一登记在 [templates/evidence_index.csv](./templates/evidence_index.csv)。

字段说明：

| 字段 | 说明 |
| --- | --- |
| evidence_id | 证据编号，如 E001 |
| date | 证据产生日期 |
| evidence_type | ai_screenshot / raw_answer / interview_note / interview_quote / report_version / public_source |
| related_file | 关联表格或文档 |
| related_record_id | 对应 run_id、interview_id 或 report_id |
| store_id | 关联门店 |
| interview_id | 关联访谈 |
| platform | AI 入口或来源平台 |
| artifact_path | 本地证据文件路径 |
| source_url_or_context | 公开 URL 或测试上下文 |
| summary | 证据摘要 |
| used_in | 用于哪份报告或判断 |
| verification_status | 待核验 / 已核验 / 已脱敏 / 不可外发 |
| notes | 备注 |

## AI 测试证据规则

每条进入报告的关键 AI 发现，至少应有：

1. [templates/answer_runs.csv](./templates/answer_runs.csv) 中的一条记录。
2. `raw_answer` 完整原文。
3. `screenshot_path` 指向本地截图。
4. AI 入口、时间、设备、账号状态、联网/搜索、定位权限、城市/商圈。
5. 如果是关键结论，至少隔 24 小时复测一次。

可以进入商家报告的证据：

- 多个 AI 入口出现同类问题。
- 同一个关键 prompt 复测后仍出现类似问题。
- 明确信息错误，如地址、人均、营业时间、项目/菜品错误。
- 竞品被推荐的理由能转成内容、团购、达人或资料任务。

只做内部参考的证据：

- 单个 AI 入口一次性回答。
- 无截图或无原始回答。
- 模型明确说无法获取实时信息。
- 只能说明排名波动，不能转成经营动作。

## 访谈证据规则

每个有效访谈至少应有：

1. [templates/interview_tracker.csv](./templates/interview_tracker.csv) 中的一条记录。
2. 关键原话，尽量保留受访者原始表达。
3. 角色和业务类型：负责人、一线运营、商家、达人/BD。
4. 是否愿意试讲、是否愿意给真实门店、是否愿意付费或说出价格。
5. 预算来源和替代竞争顾虑。
6. 对应的下一步动作。

强信号必须满足至少一条：

- 愿意提供真实门店。
- 愿意拿样例给商家试讲。
- 愿意付费试点或说出明确可接受价格。
- 能明确说出它应进入签约、月报、续费或投放前诊断。

弱信号：

- 只说概念有趣。
- 不知道怎么卖给商家。
- 觉得和现有工具重复。
- 不愿意提供真实门店。

## 最终进入判断所需证据包

要把 goal 标记完成，最低证据包建议包括：

1. `templates/interview_tracker.csv`：至少 5 个有效服务商访谈。
2. [27_interview_synthesis_template.md](./27_interview_synthesis_template.md)：汇总强/弱/反对信号。
3. `templates/answer_runs.csv`：至少 1 家真实门店、20 个 prompts、3 个 AI 入口。
4. `templates/report_scorecard.csv`：由真实 AI 回答生成。
5. 至少 1 份真实门店报告，基于 [13_real_store_report_template.md](./13_real_store_report_template.md)。
6. `templates/evidence_index.csv`：所有截图、原始回答、访谈原话、报告版本均有索引。
7. 更新 [00_decision_memo.md](./00_decision_memo.md) 和 [15_final_research_synthesis.md](./15_final_research_synthesis.md)，明确 Go、No-Go 或继续小规模验证。

## 审计口径

最终不要只问“文件有没有写完”，而要逐项确认：

| 问题 | 需要的证据 |
| --- | --- |
| 国外工具是否有参考价值 | 01、23 中的工具对照和来源 |
| 国内服务商是否有需求 | 真实访谈记录和 27 汇总 |
| 市场是否值得继续 | 25 的模型 + 真实付费/预算反馈 |
| 竞争是否可避开 | 24 的替代竞争扫描 + 访谈反馈 |
| 技术是否可实现 | 05、26 + 真实跑数成本 |
| 产品闭环是否成立 | 服务商是否愿意试讲/放进月报/续费 |
| 产品样例是否可信 | 真实门店报告 + answer_runs + 截图索引 |
